# mareumi-flash
国服英雄联盟AI插件
这是我 vibecoding 以来第三个项目。之前做了两个，一个是 WinForm + SQLite 的排班系统，一个是服务大队的 H5 + React + Node.js + MongoDB 的举报上报系统。
排班系统给内勤了，也没跟踪有没有在用，其实做出来我也感觉对实际生产提升不大，最多是把每天 10 分钟的工作压缩到 1 分钟了。第二个呢，因为我不在单位，项目能不能落地生产还是个未知数。写在这里，是想借着 Mareumi 一并做一次复盘。

起点
上班的时候刷到一些 vibecoding 相关的视频，就想我能不能做这个事情？然后开始拼命从工作中挖掘需求，这一路的尝试就这么开始了。
排班系统是问 AI 问出来的——是 AI 提供的方案里的一个切入点，我就从这个点开始做。做起来比较顺，以前在公司也用过 AI，但那时候还停留在复制粘贴代码。这次不一样的地方，我觉得是 Agent 架构成熟了，AI 从聊天工具真正走向了生产。
以前如果我要做排班系统，可能要搜几周：前端怎么写、数据库怎么连……AI 帮我省掉了大量学习成本。做出来之后的感觉不是"哇我好厉害"，而是"我本来就应该能做这个"——工具到位了，挡在前面的墙就不见了。

排班系统
2 天做完。
踩的坑主要是模型选型。前端有个问题，用 Haiku 4.5 怎么搞都不对，换 Opus 一次就解决了。后来想明白了：前端问题算是最基础的任务，如果模型搞不定，说明能力边界就在那了，与其死磕不如直接换。
后端逻辑问题是在 GitHub Copilot 里搞的，必须把整个逻辑清清楚楚梳理给它，才能得到一个能用的结果——沟通成本很高。这和前端问题的性质不一样：前端是模型能力到头了，后端是你得先想清楚再喂给它。
总结下来就一条：项目复杂度上去了，弱模型省的那点钱，会用更高的沟通成本还回来。

上报系统
做出来排班系统，觉得要搞点更实际的，把自己手头的活用程序解放出来，就做了上报系统。
初期选的是微信小程序 + 云开发，也是两天做出来了，基本跑得通。但小程序测试版最多只能 15 个用户，放在真实的业务场景里不够用。想着审核就审核吧，在 IDE 里花了 2 天专门优化合规——结果因为个人主体小程序不能提供视频服务，直接被打回来。
平台政策的硬墙，绕不过去，只能换技术栈。整体迁到 H5，部署到腾讯云服务器，直接浏览器访问，审核问题就彻底没了。
迁移的过程反而挺顺的，因为这时候工作流也升级了：Claude 出方案，Cursor 落实。Claude 给出细致的方案，Cursor 执行起来就顺很多。搭脚手架、迁前端、迁后端，基本一气呵成，没踩什么技术坑。比之前在 Copilot 里又想方案又写代码、反复解释上下文，省了不少心。
这个项目最大的教训不是技术，是选型要早点考虑平台的硬限制——合规风险、政策边界，要在一开始就评估，不然后面做的优化都可能是无效工作。

Mareumi
这是这次复盘的主角。
从桌宠到插件
最开始想做的是桌宠，有个游戏形象陪着打游戏。结果卡在了设计上——手头没有好用的像素形象设计工作流，做出来的东西一直不满意。卡了一段时间之后，决定把方向改成对话插件。这个转变不是技术决策，是被一个设计瓶颈推着走的。回头看不算坏事，插件反而更干净。
视觉识别：走了弯路，砍掉
想做局内实时数据采集，试了模型识别和 OCR 两条路。固定 HUD 的信息还好，英雄、装备、技能这类识别效果很差——图标多、皮肤干扰大，OCR 基本没戏，模型识别又贵又慢。最后把视觉识别整体砍掉，数据来源收窄到 Riot API，只抓选人阶段的数据。
砍掉之后范围窄了，但能做扎实。
语音 + 提示音
语音识别和 TTS 一开始就定了方向，接进去没踩什么坑。倒是有个小细节值得记：最初的交互是按键说话，用着感觉无感，不知道系统在干什么。后来加了提示音——按键、识别中、模型思考、工具调用，每个阶段都有声音反馈。
这个改动没有人提需求，是自己用的时候感觉到"差点意思"主动加的。这种判断从哪来？用别的产品用多了，内化了什么是好的体验，自然就往那个方向靠。
最终形态
Riot API 拿选人阶段数据 + 语音识别 + TTS + 对话插件。跑通了。

回头看
关于工具
从在 Copilot 里又想又写、到 Claude 出方案 Cursor 落实，这个工作流的升级是这几个项目里最实在的收获。方案想清楚了，执行才顺。
关于模型
复杂度上去了，就得用更强的模型，这是绕不过去的事。前端问题可以当成模型能力的金丝雀——搞不定就换，别死磕。
关于做中学
好奇心有，但不喜欢啃理论啃底层，更偏向宏观理解。知道大模型是概率模型，但不会深挖输入到输出的实现细节。比起彻底吃透，更在意的是"理解到能做决策的深度"。
有时候会担心，这种理解程度和彻底吃透的人做出来的东西会不会有差距。目前还没有遇到真正被底层知识卡住的时刻——因为一直有意识地把实现限定在自己能掌控的范围里。这个焦虑可能不是来自真实的差距，而是来自不知道自己的上限在哪。
关于放弃和搁置
如果某个功能 AI 不知道怎么做、自己也不知道怎么做，那就先放下。不是放弃，是等工具或者条件成熟了再捡。上报系统的视频功能就是这样——小程序做不了，换了个能实现的载体，跑通了。
关于掌控感
这种创造性的工作做起来比打游戏还上瘾。说不清楚为什么，大概是因为有一种掌控感——所有的决策都过自己这一关，自己说了算。这和打工是反的。
