# Mareumi-Flash
国服《英雄联盟》AI插件——探索用技术解放生产力

## 项目背景
Mareumi-Flash 是我在 vibecoding 旅程中的第三个项目。第一次尝试的项目是一个基于 WinForm 和 SQLite 的排班系统；第二次是一个服务大队的举报上报系统，结合了 H5、React、Node.js 和 MongoDB。通过这些项目，我逐渐总结出一些经验与教训，使得这个项目得以顺利落地。

### 排班系统
- **开发周期**：2天
- **技术细节**：前端耗时主要集中在模型选择上。Haiku 4.5 并不适配开发需求，更换为 Opus 后一次性解决。后端逻辑利用 GitHub Copilot 搞定，然而沟通成本较高，需要详细梳理逻辑。
- **心得**： 
   1. 对于复杂度较高的项目，模型的选型尤为重要。
   2. 弱模型虽然成本低，但可能提高沟通和开发时间成本。

### 举报上报系统
- **初版开发**：基于微信小程序和云开发。
- **问题与改进**：测试版最多支持15位用户，难以满足真实业务需求。此外，小程序审核造成效率受限。最终选择迁移至 H5 并部署到腾讯云。
- **技术工作流**：Claude 用于设计方案，Cursor 负责实现细节，显著提高了开发效率。
- **重要教训**：技术选型需快速评估平台的政策和合规环境，以避免后期重复开发。

## Mareumi 的故事
- **最初设想**：尝试制作一款桌宠，让一个游戏形象与玩家“共同作战”。
- **设计卡点**：因缺乏像素图设计的清晰工作流，项目在初期遇阻。

### 视觉识别：试验与精简
- **尝试**：开发游戏局内实时数据采集功能。
- **挑战**：英雄、装备与技能数据的识别效果受限于复杂的图像环境（如图标多样、皮肤干扰）。
- **策略调整**：放弃视觉信息识别，转而专注于更清晰具体的功能。

### 音频交互：语音识别与 TTS
- **起点**：语音识别与文本转语音功能早早确定为重点方向。
- **用户体验**：早期交互依赖按键触发语音指令，效果平平。后续优化为实时语音提示增强用户反馈。
- **改进来源**：没有人提出这项需求，改进完全源于是自己使用时的感受和经验判断。

### 最终形态
通过 Riot API 获取选人数据，结合语音识别与 TTS 实现高度集成的对话插件，成功完成开发。

## 项目复盘
### 关于工具
从 Copilot 的一体化开发到利用 Claude + Cursor 的方案化工作流，是几次项目中最突出的效率提升。

### 关于模型
随着项目复杂度增加，必须选择更强大的模型。若前端问题暴露模型能力不足，应果断更换模型，避免浪费开发时间。

### 关于学习方式
相比深入底层原理，更注重宏观理解与应用。这种选择尚未对开发效率或结果造成不良影响。

### 关于搁置与放弃
功能开发中若遇到自己与 AI 均无法解决的问题，不排除暂时搁置，待条件成熟时重新拾起。例如，上报系统的视频功能改进计划就是如此。

### 关于掌控感
项目开发的创造性工作带来了强烈的满足感，这种掌控感远超日常打工体验。
